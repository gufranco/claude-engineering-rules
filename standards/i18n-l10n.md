# Internationalization and Localization

## Core Distinction

- **i18n** (internationalization): design the software so it can be adapted to different languages and regions without code changes
- **l10n** (localization): adapt the software for a specific language, region, or culture

i18n is an engineering task. l10n is a content task. Do i18n first, then l10n becomes a translation workflow.

## Text Externalization

Never hardcode user-facing strings in code. Extract them to resource files.

| Framework | Format | Library |
|-----------|--------|---------|
| React | JSON | react-intl, react-i18next |
| Next.js | JSON | next-intl, next-i18next |
| Node.js backend | JSON | i18next |
| Go | JSON or TOML | go-i18n |
| Python | PO/MO or JSON | gettext, Babel |

Rules:
- One resource file per locale: `en-US.json`, `pt-BR.json`, `es-MX.json`
- Key naming: dot-separated namespace path. `order.status.shipped`, not `shippedStatusText`
- Never concatenate translated strings. "Hello " + name + ", welcome" breaks in languages where word order differs. Use interpolation: `t('greeting', { name })`
- Never assume sentence structure. Plurals, gender, and word order vary across languages

## Pluralization

Different languages have different plural rules. English has 2 forms (singular, plural). Arabic has 6. Russian has 3.

Use ICU MessageFormat or the library's built-in plural support:

```json
{
  "items": "{count, plural, =0 {No items} one {# item} other {# items}}"
}
```

Never use conditional logic like `count === 1 ? 'item' : 'items'`. It fails for any language with more than two plural forms.

## Number, Date, and Currency Formatting

Use `Intl` APIs or the equivalent locale-aware formatting library.

| Data type | Formatter | Example |
|-----------|-----------|---------|
| Numbers | `Intl.NumberFormat` | `1,234.56` (en-US) vs `1.234,56` (de-DE) |
| Currency | `Intl.NumberFormat` with `style: 'currency'` | `$1,234.56` vs `R$ 1.234,56` |
| Dates | `Intl.DateTimeFormat` or date-fns `format` with locale | `3/31/2026` vs `31/03/2026` |
| Relative time | `Intl.RelativeTimeFormat` | "2 hours ago" vs "hace 2 horas" |

Never format numbers or dates manually with string templates.

## RTL (Right-to-Left) Support

Languages like Arabic, Hebrew, and Persian read right-to-left. The layout must mirror.

- Use CSS `direction: rtl` and `writing-mode` on the root element, toggled by locale
- Use logical CSS properties: `margin-inline-start` instead of `margin-left`, `padding-inline-end` instead of `padding-right`
- Icons with directional meaning (arrows, progress bars) must flip in RTL
- Test with actual RTL text, not just the `dir` attribute. Mixed content (English words in Arabic text) creates bidirectional text that needs `<bdi>` or `unicode-bidi` handling

## Locale Detection

| Source | Priority | When |
|--------|----------|------|
| User preference (profile setting) | Highest | Authenticated users |
| URL parameter or path (`/en-US/...`) | High | Public pages with SEO requirements |
| `Accept-Language` header | Medium | First visit, no preference set |
| Browser `navigator.language` | Low | Client-side fallback |
| Default locale | Lowest | When nothing else is available |

Store the resolved locale in the user's session or profile. Do not re-detect on every request.

## Translation Workflow

- Source language is the default locale. Translators work from source to target
- Use a translation management system (Crowdin, Lokalise, Phrase) for professional projects
- Export translations as JSON or PO files checked into the repository
- CI validates that all locales have the same keys. Missing keys are build failures
- Never machine-translate and ship without human review

## Testing

- Render every screen in at least two locales (default + one with longer text, like German)
- Verify that long translations do not break layout (German text is typically 30% longer than English)
- Test pluralization with 0, 1, 2, 5, and 21 (to catch Russian-style rules)
- Test RTL layout with Arabic or Hebrew if the application supports RTL
- Use pseudo-localization in development: replace characters with accented versions to catch hardcoded strings

## Regional Variants

Languages spoken in multiple countries have vocabulary, spelling, and grammar differences. "Color" vs "colour", "onibus" vs "autocarro", "computadora" vs "ordenador". Using the wrong regional variant is a localization bug.

See `standards/i18n-regional-variants.md` for the full standard: flagged word lists, dictionary-based verification, and CI integration
